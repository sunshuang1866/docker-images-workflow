# CI 失败分析报告

## 基本信息
- PR: #2568 — 【自动升级】slurm容器镜像升级至-26-05-1-1版本.
- 失败类型: build-error
- 置信度: 高
- 知识库匹配: 新模式
- 新模式标题: Docker标签非法前缀
- 新模式症状关键词: `invalid tag`, `invalid reference format`, leading hyphen, `docker.io/`

## 根因分析

### 直接错误
```
ERROR: invalid tag "docker.io/****test/slurm:-26-05-1-1-oe2403sp3-x86_64": invalid reference format
```

### 根因定位
- 失败位置: CI 构建流水线（eulerpublisher 镜像检查阶段）
- 失败原因: 镜像版本号 `-26-05-1-1` 以连字符 `-` 开头，CI 系统在基于目录路径构造 Docker image tag 时生成了 `-26-05-1-1-oe2403sp3-x86_64` 这样的非法 tag（Docker tag 不允许以 `-` 开头），导致 `docker buildx` 拒绝该引用格式。

### 与 PR 变更的关联
**直接关联。** PR 新增了目录 `Others/slurm/-26-05-1-1/24.03-lts-sp3/Dockerfile`：版本号以 `-` 开头的路径名（如 `-26-05-1-1/`）非标准语义。CI 的 eulerpublisher 工具从目录层级中提取版本号并拼接 image tag，版本字符串首字符即为 `-`，导致最终 tag 以 `-` 开头，违反 Docker 的 tag 命名规范（只能以字母数字或下划线开头）。

`meta.yml` 中同样使用了以 `-` 开头的 key：
```yaml
-26-05-1-1-oe2403sp3:
  path: -26-05-1-1/24.03-lts-sp3/Dockerfile
```

Dockerfile 内的 `ARG TAG=slurm-25-11-6-1` 与目录命名无关，不影响本次失败。

## 修复方向

### 方向 1（置信度: 高）
版本号 `-26-05-1-1` 中的前导连字符不符合 Docker tag 命名规范。需要确认这个版本号的上游来源是否正确 — `-26-05-1-1` 很可能是一个包含拼写或格式错误的版本号（正常 Docker tag 不应以 `-` 开头）。确认正确版本后，将目录名、文件路径、`meta.yml` 中的版本号修正为不以 `-` 开头的合法格式。

### 方向 2（置信度: 中）
如果 `-26-05-1-1` 确实是上游的正确版本标识符，则需要在 CI 工具链（eulerpublisher）层面增加 tag 合法性预处理，例如在拼接 image tag 前将前导 `-` 替换为合法前缀（如 `v-26-05-1-1` 或 `slurm-26-05-1-1`）。

## 需要进一步确认的点
- 上游 Slurm 仓库是否存在版本号为 `-26-05-1-1` 的 git tag，还是自动升级脚本因版本号格式异常生成了错误路径。
- 自动升级脚本提取版本号的逻辑是否需要处理版本字符串前缀非法字符的边界情况。
- 其他已存在的 Slurm Dockerfile（如 `25.11.6/`）路径中版本号均不以 `-` 开头，`-26-05-1-1` 是首个此类案例，需确认是否为自动生成工具的参数传递错误。
