# 修复摘要

## 修复的问题
修复 `fix_getdeps.py` 中 `_verify_hash` 方法替换的正则表达式边界匹配缺陷：当 `_verify_hash` 是类中最后一个方法时，前瞻断言无法匹配，导致替换静默失败。

## 修改的文件
- `Others/fbthrift/2026.06.08.00/24.03-lts-sp3/fix_getdeps.py`: 修改第 17 行正则表达式，在前瞻断言中增加 `\n\S` 和 `\Z` 两种替代边界

## 修复逻辑
原正则 `r'def _verify_hash\(self\):.*?(?=\n    def )'` 使用「下一个同缩进级别的 `def `」作为前瞻边界。当 `_verify_hash` 是类中最后一个方法时，后续不存在同缩进的 `    def `，前瞻无法匹配，`re.sub` 不会执行任何替换，导致 libaio 哈希校验未被跳过，构建可能因哈希不匹配而失败。

修改后的正则 `r'def _verify_hash\(self\):.*?(?=\n    def |\n\S|\Z)'` 增加了两种边界：
- `\n\S`：匹配行首非空白字符（0 缩进的顶层结构，如下一个 `class`、`def` 等），作为类结束的标识
- `\Z`：匹配字符串结尾，处理 `_verify_hash` 是文件中最后一个方法的情况

## 潜在风险
- 如果 `fetcher.py` 中在 `_verify_hash` 方法之后、下一个 0 缩进结构之前存在有意义的 0 缩进注释，这些注释会被连带替换掉。但鉴于目标文件是自动生成的 getdeps 代码，此风险可忽略。
- 日志截断导致无法确认本次 CI 失败是否由此正则缺陷直接引起，但该修复纠正了一个明确存在的代码缺陷。
